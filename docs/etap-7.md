# Этап 7: массовая проверка (`check --all`) и табличный отчёт

Документ фиксирует, **что реально сделано** на этом этапе в репозитории Range Program. Цель этапа: **одной командой** прогонять проверку по **всем монетам** из хранилища, выводить **сводную таблицу** в терминале и **краткую статистику** по статусам — **без** новой математики диапазонов, **без** изменений **`RangeEngine`** и **`Evaluator`**, **без** backtest и **без** уведомлений.

Связанные документы: общий план — [`plan.md`](plan.md); этап 6 (оценка одной монеты) — [`etap-6.md`](etap-6.md).

---

## Цель этапа

1. Добавить **`range check --all`** (и короткий **`-a`**) для проверки **всех** монет из **`data/coins.json`**.
2. Сохранить **`range check SYMBOL`** для подробного вывода по одной монете (как в этапе 6).
3. Вывести **таблицу** с ключевыми колонками и **Summary** в конце.
4. При сбое по одной монете **не прерывать** весь прогон — строка с **ERROR** и кратким текстом.

---

## Что сделано

### 1. Модуль отчёта и строка таблицы

Файл: **`src/range_program/check_all_report.py`**.

- **`CheckTableRow`** — строка отчёта: символ, цена, активный и рекомендуемый диапазоны в коротком виде (**`59000–76000`**), три процента (**DIST↓**, **DIST↑**, **DEV** — с округлением до **одного** знака после запятой), **STATUS**.
- **`CheckTableRow.from_check_result(CheckResult)`** — заполнение из уже посчитанного **`CheckResult`** (те же метрики, что и раньше).
- **`CheckTableRow.error_row(symbol, message)`** — для сбоев: в колонках метрик **`—`**, в **DEV** — укороченное сообщение, **STATUS** = **`ERROR`**.
- **`STATUS_SORT_ORDER`** / **`status_sort_key`** — числовой приоритет для сортировки: сначала более «проблемные» статусы (**`OUT_OF_RANGE`**, затем **`ERROR`**, **`REPOSITION`**, **`STALE`**, **`WARNING`**, **`OK`**).
- **`print_check_all_table`** — выравнивание колонок по ширине содержимого, заголовок без цвета, строки с **цветом** по статусу через **`typer.secho`** (красный / пурпурный / жёлтый / белый / зелёный в зависимости от статуса; **`ERROR`** — красный).
- **`aggregate_counts`** / **`print_summary`** — подсчёт и вывод блока **Summary** по шести категориям (включая **ERROR**), нули показываются явно.

Новые зависимости (**`rich`** и т.п.) **не** добавлялись.

---

### 2. Расширение `CheckService`

Файл: **`src/range_program/services/check_service.py`**.

- **`run_check_safe(symbol)`** — оборачивает существующий **`run_check`** в **`try`/`except ValidationError`**: возвращает **`(CheckResult, None)`** при успехе или **`(None, сообщение)`** при ошибке. Логика **recalc / цена / Evaluator / сохранение `last_check`** не дублируется.
- **`run_check_all()`** — для каждой монеты из **`list_coins()`** вызывает **`run_check_safe`**, строит **`CheckTableRow`**, сортирует по **`(sort_rank, symbol)`**.

---

### 3. CLI

Файл: **`src/range_program/cli.py`**.

Команда **`check`** объединяет два режима:

| Вызов | Поведение |
|--------|-----------|
| **`range check BTC`** | Один тикер: подробный текст (как в этапе 6), при ошибке — выход с кодом **1**. |
| **`range check --all`** или **`range check -a`** | Все монеты: таблица + **Summary**; ошибки по отдельным монетам только в строках **ERROR**. |
| Одновременно **SYMBOL** и **`--all`** | Сообщение об ошибке и код **1**. |
| Ни **SYMBOL**, ни **`--all`** | Подсказка и код **1**. |

Пустой список монет: сообщение о том, что проверять нечего, и **Summary** с нулями.

---

## Как устроен цикл `--all`

1. **`CheckService.list_coins()`** (через репозиторий) — полный список монет.
2. Для **каждой** монеты: **`run_check_safe(symbol)`** → внутри тот же **`run_check`**, что и для одной монеты (при необходимости **recalc**, цена, **Evaluator**, запись **`last_check`**).
3. Успех → строка из **`CheckResult`**; ошибка → строка **`error_row`**.
4. Сортировка таблицы: **приоритет статуса** (см. **`STATUS_SORT_ORDER`**), затем **алфавит по символу**.

Новых алгоритмов оценки нет — только агрегирование и отображение.

---

## Что намеренно не входило в этап

- Изменение **`Evaluator`**, **`RangeEngine`**, **`RecalcService`** по смыслу расчётов.
- Отдельные уведомления (Telegram, почта и т.д.).
- Backtest и «исторические» отчёты.

---

## Итог этапа

Появился **ежедневный сценарий**: **`range check --all`** даёт **одним взглядом** картину по портфелю монет (таблица + сводка), при этом **та же бизнес-логика**, что и у **`range check SYMBOL`**, вызывается через **`run_check`** без дублирования. Это опирается на этап 6 и готовит почву для дальнейших улучшений по [`plan.md`](plan.md) (журналы, экспорт, фильтры — отдельными задачами).
