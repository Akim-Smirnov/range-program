# Этап 12: несколько бирж (ccxt), fallback по котировкам и кэш рынка

Документ фиксирует, **что реализовано** в репозитории на этом этапе. Цель: убрать жёсткую привязку данных к одной бирже (по умолчанию Binance + USDT), чтобы монеты, которых нет на первой площадке, всё равно получали **цену** и **свечи** через **автоматический поиск рабочей пары** на альтернативных биржах и с альтернативными котируемыми активами.

Связанные документы: [`plan.md`](plan.md) (этап 4 — первичное подключение рынка); этап 11 — [`etap-11.md`](etap-11.md) (капитал и сетка; не пересекается по смыслу).

**Принцип архитектуры:** вся логика выбора биржи/пары сосредоточена в **`MarketDataService`**; CLI только вызывает сервис и показывает результат; **`RangeEngine`**, **`Evaluator`** и репозиторий не дублируют fallback.

---

## Проблема до этапа

- Цена и OHLCV для монеты из хранилища фактически опирались на один источник (дефолт в конфиге: обычно Binance, пара `BASE/USDT`).
- Тикеры вроде **HYPE** могут отсутствовать на Binance, но торговаться на Bybit/OKX и т.д. — программа не находила рынок без ручных обходных путей.

---

## Цель этапа

1. **Несколько бирж через ccxt** с фиксированным порядом для MVP: `binance` → `bybit` → `okx` → `kucoin` (константы в конфиге).
2. **Fallback по котируемому активу:** `USDT` → `USDC` → `USD` (с учётом явного предпочтения у монеты).
3. **Предпочтения в модели монеты:** опциональные `exchange` и `quote_asset` — пробуются первыми в общем порядке обхода.
4. **Единый резолвер рынка** для цены и свечей: не копировать перебор в `get_current_price` и `get_ohlcv` отдельно.
5. **Кэш найденного рынка** в `Coin`: `resolved_exchange`, `resolved_symbol_pair`, `resolved_at` — чтобы не искать заново при каждом вызове (с повторной проверкой тикером при наличии кэша).
6. **Понятная ошибка**, если рынок нигде не найден (без длинного traceback в CLI).
7. **Обратная совместимость JSON:** новые поля опциональны; старые `coins.json` читаются без миграции.

---

## Что сделано

### 1. Конфигурация

Файл: **`src/range_program/config.py`**.

- **`FALLBACK_EXCHANGES`** — кортеж id бирж ccxt.
- **`FALLBACK_QUOTE_ASSETS`** — порядок котируемых активов для поиска пары.
- **`DEFAULT_EXCHANGE_ID`** / **`DEFAULT_QUOTE_ASSET`** — по-прежнему для режима «только символ, без монеты в хранилище» (например `range price BTC`, когда монеты нет в списке).

### 2. Модель `MarketSymbolMatch`

Файл: **`src/range_program/models/market_symbol_match.py`**.

Результат успешного разрешения: **`exchange`**, **`symbol_pair`** (например `HYPE/USDT`), **`quote_asset`**.

Экспорт из пакета моделей: **`src/range_program/models/__init__.py`**.

### 3. Расширение `Coin`

Файл: **`src/range_program/models/coin.py`**.

| Поле | Назначение |
|------|------------|
| **`exchange`** | Предпочтительная биржа (ccxt id) или `null` — только общий fallback. |
| **`quote_asset`** | Предпочтительная котировка или `null` — сначала общий список USDT/USDC/USD. |
| **`resolved_exchange`** | Последняя успешно использованная биржа. |
| **`resolved_symbol_pair`** | Полная пара, например `HYPE/USDT`. |
| **`resolved_at`** | Время фиксации кэша (UTC). |

**`Coin.create`** принимает новые поля опционально.

### 4. `MarketDataService`

Файл: **`src/range_program/services/market_data.py`**.

- Ленивое создание и кэш инстансов ccxt: **`_get_exchange(exchange_id)`**.
- **`resolve_market(coin) -> MarketSymbolMatch`** — единая точка поиска:
  1. Если заданы **`resolved_*`**, сначала проверка тикером по сохранённой паре.
  2. Иначе вложенные циклы: **сначала котируемый актив** (с учётом preferred), **внутри — биржи** (с учётом preferred). Так для каждого quote перебираются все площадки (типичный сценарий: «сначала везде USDT, потом везде USDC»).
  3. Успех при первом непустом **`fetch_ticker`** по паре (внутренне **`_try_fetch_ticker`** глотает ошибки ccxt и возвращает `None`).
- **`get_price_quote` / `get_current_price` / `get_ohlcv`**: при переданном **`coin`** используется **`resolve_market`** и затем **`fetch_price_quote_with_match`** / **`fetch_ohlcv_with_match`**. Без **`coin`** — прежнее поведение только с дефолтной биржей и дефолтной котировкой.
- **`pair_for_symbol(symbol, coin=None)`**: если у монеты есть **`resolved_symbol_pair`**, отображается она.

### 5. Сервисы, использующие рынок по монете

- **`RecalcService`** — один раз резолвит рынок, грузит OHLCV и цену, при успехе обновляет **`resolved_*`** в сохранённой монете.
- **`CheckService`** — то же для проверки цены и обновления **`last_check`**.
- **`backtest.run_backtest`** — передаёт **`coin`** в **`get_ohlcv`**.

### 6. JSON и `CoinRepository`

Файл: **`src/range_program/repositories/coin_repository.py`**.

Сериализация/десериализация новых полей; отсутствующие ключи в старых записях дают **`None`**.

### 7. `CoinService`

Файл: **`src/range_program/services/coin_service.py`**.

**`add_coin(..., exchange=, quote_asset=)`** прокидывает параметры в репозиторий.

### 8. CLI

Файл: **`src/range_program/cli.py`**.

| Изменение | Описание |
|-----------|----------|
| **`range add SYMBOL --exchange / -e`** | Сохранить предпочтительную биржу. |
| **`range add SYMBOL --quote / -q`** | Сохранить предпочтительную котировку. |
| **`range resolve-market SYMBOL`** | Взять монету из хранилища, вызвать **`resolve_market`**, вывести биржу/пару/quote и **записать** **`resolved_*`**. |
| **`range show SYMBOL`** | Дополнительно: preferred exchange/quote, resolved exchange/pair/at; блок последней цены через **`get_price_quote(..., coin=coin)`**. |
| **`range price`**, **`range candles`**, **`recalc`**, **`check`**, **`backtest`**, **`optimize`** | Для монет из списка данные идут через новый механизм (**`coin`** в **`MarketDataService`**), без дублирования fallback в CLI. |

Ошибки **`MarketDataError`** в CLI по-прежнему краткие (сообщение пользователю, без стека).

## Что намеренно не входило в этап

- Глобальная предзагрузка рынков всех бирж при старте CLI.
- Отдельный persistent-кэш рынков вне **`coins.json`**.
- Сложная конфигурация списков бирж через пользовательский файл (достаточно констант в **`config.py`** для MVP).
- Торговые операции на бирже — только **чтение** котировок и свечей.

## Связь с `plan.md`

В [`plan.md`](plan.md) в разделе про этапы заложена идея «сначала одна биржа, затем абстракция под несколько» — этап 12 реализует **практический слой нескольких источников** для **данных**, не меняя математику диапазона.

---

## Итог этапа

Программа **автоматически находит** рабочую пару на одной из поддерживаемых бирж и с одной из котировок USDT/USDC/USD, **запоминает** успешный вариант в монете, и все команды, завязанные на **`MarketDataService`** с **`Coin`**, используют этот механизм **централизованно**. Зависимость только от Binance+USDT для монет из списка снята.
