# Этап 3: расширение модели данных и хранилища

Документ фиксирует, **что реально сделано** на этом этапе в репозитории Range Program. Цель этапа: **подготовить полноценную модель и JSON-хранилище** под будущую логику диапазонов — **без расчёта диапазонов, без рыночных API и без СУБД** (только файл `data/coins.json`).

Связанные документы: общий план — [`plan.md`](plan.md); этап 2 (каркас) — [`etap-2.md`](etap-2.md).

---

## Цель этапа

Перейти от «монета = тикер + дата создания» к **расширенной карточке монеты** с параметрами анализа (пока только хранение), к **активному диапазону** как отдельной сущности, к **предсказуемому API репозитория** и к **обновлённому CLI** с командами установки и сброса активного диапазона.

---

## Что сделано

### 1. Модель `Coin`

Файлы: **`src/range_program/models/coin.py`**, константы по умолчанию — **`src/range_program/models/defaults.py`**.

Поля:

- **Обязательные (как и раньше):** `symbol`, `created_at`.
- **Новые поля под будущую аналитику:** `mode` (`conservative` / `balanced` / `aggressive`), `timeframe` (строка, напр. `4h`), `lookback_days` (int), `center_method`, `width_method`, `updated_at`.
- **Вложенное поле:** `active_range` — может быть `null` (нет активной сетки в данных).

Фабрика **`Coin.create(...)`** задаёт дефолты для новой монеты:

- `mode = balanced`
- `timeframe = 4h`
- `lookback_days = 60`
- `center_method = ema`
- `width_method = atr`

### 2. Модель `ActiveRange`

Файл: **`src/range_program/models/active_range.py`**.

Поля: `low`, `high` (float), `set_at` (datetime), `comment` (опционально, строка или отсутствует).

Назначение: хранить **факт** «где пользователь поставил сетку», отдельно от будущего **рекомендуемого** диапазона (это по-прежнему вне этапа 3).

### 3. JSON-хранилище

Файл: **`src/range_program/repositories/coin_repository.py`**.

- Структура **`data/coins.json`**: массив объектов с полной моделью монеты и вложенным `active_range` или `null`.
- Даты сериализуются в **ISO 8601** (UTC).
- **Пустой файл** и отсутствие файла обрабатываются**:** создаётся каталог `data/`, при необходимости — пустой массив `[]`.
- **Миграция со старого формата:** записи только с `symbol` и `created_at` (этап 2) при чтении дополняются дефолтными полями; `updated_at` при отсутствии берётся из `created_at`.
- **Защита от поломанных данных (минимально):** невалидный JSON при загрузке даёт понятную ошибку; при разборе каждой записи ошибки приводят к сообщению с индексом элемента.

### 4. API репозитория `CoinRepository`

Вся работа с файлом изолирована в репозитории. Методы:

- **`add_coin(symbol, ...)`** — параметры по умолчанию или переопределённые; дубликат по символу не добавляется.
- **`remove_coin(symbol)`**
- **`get_coin(symbol)`**
- **`list_coins()`**
- **`update_coin(coin)`** — замена записи с тем же `symbol`.

### 5. Сервисный слой

Файл: **`src/range_program/services/coin_service.py`**.

- Операции **`set_active_range`** и **`clear_active`**: проверка существования монеты, обновление `active_range` и `updated_at` через **`update_coin`**.
- Валидация режима и границ диапазона вызывается до записи (см. ниже).

### 6. Валидация

Файл: **`src/range_program/validation.py`**.

- Режим: только значения из **`ALLOWED_MODES`** (согласовано с `defaults.py`).
- Активный диапазон: **`low < high`**.
- Ошибки домена: исключение **`ValidationError`** с текстом; CLI переводит это в сообщение пользователю и ненулевой код выхода.

Дополнительно по требованиям этапа: **тикер в верхний регистр**, **нельзя дублировать монету при add**, **нельзя выполнять операции по несуществующей монете** там, где это указано (например `set-active` / `clear-active`).

### 7. Обновление CLI

Файл: **`src/range_program/cli.py`**.

- **`range add`** — позиционный `symbol` и опции: `--mode`, `--timeframe`, `--lookback`, `--center-method`, `--width-method`; без опций используются дефолты выше.
- **`range show`** — вывод всех полей монеты и блока `active_range` (или явное «нет»).
- **`range list`** — компактный список (символ, режим, таймфрейм, lookback, `updated_at`).
- **`range set-active SYMBOL --low X --high Y`** — опционально **`--comment`**; `set_at` = текущее время UTC.
- **`range clear-active SYMBOL`** — сброс активного диапазона.

Команды **`remove`** и базовый сценарий **`add` / `list` / `show`** сохранены и расширены по содержимому.

### 8. Тесты

- **`tests/test_coin_repository.py`** — добавление/удаление/список, round-trip JSON, миграция legacy-записи, `update_coin`, невалидный JSON.
- **`tests/test_coin_service.py`** — отсутствующая монета для `set-active`, нарушение `low < high`.

Запуск: **`python -m pytest tests`**.

---

## Что намеренно не входило в этап

- Расчёт рекомендуемого диапазона, волатильность, «умная» аналитика.
- Подключение API бирж и загрузка котировок.
- Базы данных (SQLite и т.д.) — по-прежнему только **JSON**.
- Команды уровня **`range check`**, **`range recalc`** и прочие из [`plan.md`](plan.md), не относящиеся к хранению модели.

---

## Итог этапа

На этапе 3 данные монеты и активный диапазон **формализованы в коде и в файле**, репозиторий даёт **стабильный контракт** для следующих слоёв, CLI позволяет **задать и снять активный диапазон** без расчётной логики. Это готовая основа для этапов с рынком и расчётами по [`plan.md`](plan.md).
