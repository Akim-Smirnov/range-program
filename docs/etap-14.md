# Этап 14: расширение методов центра диапазона в `RangeEngine`

Документ фиксирует, **что реализовано** в репозитории на этом этапе. Цель: расширить **`center_method`** у монеты так, чтобы центр рекомендуемого диапазона можно было считать не только как **`price`** и **`ema`**, но и как **SMA**, **медиану**, **середину high/low окна** и **центр канала Дончиана** (MVP), **без переписывания** ширины по ATR, clamp, геометрической сетки и остального контура приложения.

Связанные документы: [`etap-5.md`](etap-5.md) (первый `RangeEngine` и смысл `center_method` / `width_method`); пользовательская сводка — [`instrukciya.md`](instrukciya.md) (раздел про параметры монеты и `center_method`).

**Принцип архитектуры:** меняется только слой **`RangeEngine.calculate_range`** (плюс валидация допустимых строк при **`add_coin`**) и точечно **CLI help**. **`width_method`**, ATR, **`Evaluator`**, **`CheckService`**, backtest, optimize, репозитории JSON и структура **`RecommendedRange`** — **без изменений поведения**, кроме новых допустимых значений поля **`center_method`** в расчёте и в сохранённой рекомендации.

---

## Проблема до этапа

- В MVP центр диапазона поддерживался только двумя способами: **`price`** (текущая котировка) и **`ema`** (EMA по закрытиям с фиксированным периодом).
- Для сравнения подходов и сценариев «центр как средняя / как медиана / как середина канала» не было **явных** режимов в коде — только ручное изменение логики или обходные трактовки.

---

## Цель этапа

1. Добавить методы центра: **`sma`**, **`median`**, **`midpoint`**, **`donchian`** рядом с **`price`** и **`ema`**.
2. Зафиксировать **отдельные константы периодов** окна (в MVP для новых методов и EMA — **20** свечей там, где используется скользящее окно; см. `range_engine.py`).
3. Вынести расчёт центра в **читаемые хелперы** (`_calculate_center_sma`, …, диспетчер **`_resolve_center`**), не раздувая один гигантский `if`.
4. Сохранить **обратную совместимость** JSON: старые записи с `center_method: ema` / `price` работают как раньше; дефолт для новой монеты — **`ema`** (`defaults.py`).
5. Ограничить допустимые значения при вводе: **`ALLOWED_CENTER_METHODS`** + **`validate_center_method`** в **`CoinService.add_coin`**.
6. Обновить **help** у `range add --center-method` и кратко **инструкцию** [`instrukciya.md`](instrukciya.md).
7. Покрыть сценарии **тестами** на синтетических свечах.

---

## Что сделано

### 1. `RangeEngine`

Файл: **`src/range_program/services/range_engine.py`**.

| Константа | Назначение (MVP) |
|-----------|------------------|
| `EMA_PERIOD` | Как раньше: EMA по ряду закрытий |
| `SMA_PERIOD`, `MEDIAN_PERIOD`, `MIDPOINT_PERIOD`, `DONCHIAN_PERIOD` | Окно **20** свечей (раздельные имена для читаемости и будущего расхождения периодов) |
| `ATR_PERIOD` | Без изменений: ширина по Wilder ATR |

**Семантика методов:**

- **`price`** — центр = **`current_price`** (как раньше).
- **`ema`** — центр = значение EMA в конце ряда закрытий (логика **`_ema_last`** сохранена).
- **`sma`** — среднее арифметическое по **последним N** закрытиям (`N = SMA_PERIOD`).
- **`median`** — медиана по **последним N** закрытиям (`statistics.median`, `N = MEDIAN_PERIOD`).
- **`midpoint`** — **(max(high) + min(low)) / 2** по **последним N** свечам (`N = MIDPOINT_PERIOD`).
- **`donchian`** — **(highest_high + lowest_low) / 2** по **последним N** свечам (`N = DONCHIAN_PERIOD`). В MVP формула совпадает с **`midpoint`** на том же окне; в коде — **отдельные функции** для возможного расхождения логики позже.

**Данные:** свечи сортируются по времени; для оконных методов берутся **последние N** элементов после сортировки.

**Минимум длины ряда:** функция **`_min_candles_for_center_method`**: для **`price`** достаточно **`ATR_PERIOD`** (14) свечей только для расчёта ATR; для остальных методов — **`max(период_центра, ATR_PERIOD)`** (фактически **20** при периоде 20). Пустой список свечей и неположительный центр — **`RangeEngineError`** с понятным текстом. Неподдерживаемый **`center_method`** — **`RangeEngineError`** (список допустимых в сообщении).

Ширина полосы (**ATR × множитель режима**), **clamp** по %, **`compute_geometric_grid_configs`** — **без изменений**.

### 2. Модель допустимых значений

Файл: **`src/range_program/models/defaults.py`**.

- **`ALLOWED_CENTER_METHODS`** — `frozenset` из шести строк: `price`, `ema`, `sma`, `median`, `midpoint`, `donchian`.
- **`DEFAULT_CENTER_METHOD`** по-прежнему **`ema`**.

Экспорт из **`src/range_program/models/__init__.py`**.

### 3. Валидация при добавлении монеты

Файл: **`src/range_program/validation.py`**.

- **`validate_center_method(center_method)`** — отклоняет значение не из **`ALLOWED_CENTER_METHODS`** с **`ValidationError`** и перечислением допустимых.

Файл: **`src/range_program/services/coin_service.py`**.

- В **`add_coin`** после **`validate_mode`** вызывается **`validate_center_method`**.

### 4. CLI

Файл: **`src/range_program/cli.py`**.

- У опции **`--center-method`** обновлён **help**: перечислены все поддерживаемые методы.

Интерактивное меню (**`menu.py`**) по-прежнему запрашивает строку **Center method** с дефолтом из **`DEFAULT_CENTER_METHOD`**; допустимость проверяется при сохранении монеты через **`CoinService`**.

### 5. Документация пользователя

Файл: **`docs/instrukciya.md`**.

- Кратко описаны допустимые **`center_method`** и отсылка к **`range_engine.py`** по периодам окна.

## Тесты

- **`tests/test_range_engine.py`**: сценарии для **`sma`**, **`median`**, **`midpoint`**, совпадение **`midpoint` / `donchian`** в MVP на одних данных, **`price`** / **`ema`** (существующие), недостаточно свечей, пустой список свечей, неизвестный **`center_method`**.
- **`tests/test_coin_service.py`**: отклонение **`add_coin`** при недопустимом **`center_method`**.

---

## Что намеренно не входило в этап

- Новые **`width_method`**, изменение ATR или множителей режима.
- Правки **`Evaluator`**, **`CheckService`**, **`backtest`**, **`optimizer`** (кроме косвенного использования того же **`RangeEngine`**).
- Разные периоды для **`midpoint`** и **`donchian`** (в MVP совпадают).
- Миграции JSON: поле **`center_method`** по-прежнему строка; неизвестное значение из ручного редактирования файла проявится при **`recalc`** на уровне **`RangeEngine`**.

---

## Обратная совместимость

- Монеты с **`center_method`: `ema`** или **`price`** ведут себя как до этапа при тех же данных.
- Для **`price`** минимальное число свечей для успешного расчёта снижено до **14** (достаточно для ATR); ранее требовалось **20** из-за общей проверки с EMA — при **≥20** свечах поведение совпадает с прежним по смыслу.

---

## Итог этапа

**`RangeEngine`** умеет считать центр рекомендуемого диапазона **шестью методами**; допустимые значения зафиксированы в **`ALLOWED_CENTER_METHODS`** и проверяются при **`add_coin`**; ширина и остальная бизнес-логика приложения **не дублировались** и **не переписывались** — выполнено точечное расширение движка и сопутствующих проверок/документации.
