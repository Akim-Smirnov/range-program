# PLAN.md

## Проект: Range Program

### Назначение

`Range Program` — это CLI-программа для расчета, хранения и контроля рабочих диапазонов сеточных ботов по нескольким монетам.

Главная задача программы:
подбирать такие диапазоны, которые:

* живут как можно дольше без ручной перестановки;
* остаются достаточно близко к текущей цене;
* не требуют постоянного вмешательства;
* позволяют быстро понять, какие сетки еще актуальны, а какие пора пересчитать или сместить.

Программа не является торговым ботом на первом этапе.
На старте это аналитический и управляющий инструмент, который:

* хранит список монет;
* рассчитывает рекомендуемые диапазоны;
* хранит активные диапазоны, по которым уже стоят боты;
* сравнивает активный диапазон с текущим рыночным состоянием;
* выдает статус и рекомендацию по каждой монете.

---

## 1. Главная идея проекта

Существует конфликт между двумя целями:

1. Диапазон сетки должен жить долго.
2. Диапазон сетки не должен слишком сильно уходить от текущей цены.

Если сделать диапазон слишком узким:

* бот будет ближе к цене;
* сетка будет эффективнее в текущем движении;
* но диапазон быстро устареет и придется часто переставлять.

Если сделать диапазон слишком широким:

* сетка проживет дольше;
* но может оказаться неэффективной, так как большая часть диапазона будет далеко от рынка.

Задача программы — найти практический баланс между этими крайностями.

---

## 2. Цели проекта

### 2.1. Основные цели

Программа должна:

* позволять добавлять монеты для анализа;
* хранить параметры анализа по каждой монете;
* получать текущую цену и исторические данные;
* рассчитывать рекомендуемый диапазон;
* показывать активный диапазон, если он уже задан пользователем;
* оценивать актуальность активного диапазона;
* подсказывать, нужно ли оставлять диапазон, наблюдать, расширять или переставлять его.

### 2.2. Практическая цель

На практике программа должна отвечать на вопрос:

> “Можно ли оставить текущую сетку как есть, или ее уже пора менять?”

И второму вопросу:

> “Если ставить новую сетку сейчас, какой диапазон выбрать, чтобы она прожила как можно дольше, но не стала слишком далекой от цены?”

---

## 3. Формат использования

Программа должна работать через терминал.

Пример сценария использования:

```bash
range add BTC
range add ETH
range list
range show BTC
range check
range recalc BTC
range recalc --all
```

На первом этапе интерфейс только CLI.
GUI и веб-интерфейс не являются приоритетом.

## 4. Границы первой версии

### Что входит в первую версию

* CLI-команды;
* хранение списка монет;
* хранение настроек по каждой монете;
* получение рыночных данных;
* расчет рекомендуемого диапазона;
* оценка текущего активного диапазона;
* вывод статуса и рекомендации;
* логирование результатов проверок.

### Что не входит в первую версию

* автоматическое выставление ордеров на бирже;

* машинное обучение;
* полностью автоматическая торговля без участия пользователя.

## 5. Ключевые принципы системы

### 5.1. Программа должна быть объяснимой

Каждое решение программы должно быть понятным:

* почему именно такой диапазон;
* по каким данным он рассчитан;
* почему статус `OK`, `WARNING` или `REPOSITION`;
* почему программа рекомендует перестановку.

Пользователь должен видеть не только итог, но и основу расчета.

### 5.2. Программа должна быть расширяемой

На старте логика может быть простой, но архитектура должна позволять:

* добавлять новые алгоритмы расчета;
* подключать новые биржи;
* добавлять новые критерии актуальности диапазона;
* тестировать разные режимы.

### 5.3. Программа должна разделять факт и рекомендацию

Есть два разных понятия:

* **активный диапазон** — тот, который пользователь реально использует;
* **рекомендуемый диапазон** — тот, который программа считает оптимальным сейчас.

Их нельзя смешивать.

## 6. Основные сущности проекта

### 6.1. Монета

Объект, который хранит:

* тикер (`BTC`, `ETH`, `SOL`);
* источник данных;
* таймфрейм анализа;
* период исторических данных;
* режим расчета;
* параметры алгоритма;
* активный диапазон;
* дату последнего пересчета.

### 6.2. Активный диапазон

Диапазон, в котором у пользователя уже работает сетка.

Содержит:

* нижнюю границу;
* верхнюю границу;
* центр диапазона;
* дату установки;
* комментарий;
* версию логики, по которой диапазон был поставлен.

### 6.3. Рекомендуемый диапазон

Новый диапазон, рассчитанный программой на текущий момент.

Содержит:

* нижнюю границу;
* верхнюю границу;
* центр;
* ширину в процентах;
* волатильность, на основе которой построен диапазон;
* причину/метод построения.

### 6.4. Проверка диапазона

Результат анализа активной сетки относительно рынка.

Содержит:

* текущую цену;
* активный диапазон;
* рекомендуемый диапазон;
* отклонение цены от центра;
* расстояние до верхней и нижней границы;
* статус;
* рекомендацию;
* timestamp проверки.

## 7. Основные команды CLI

### 7.1. Управление монетами

#### `range add <symbol>`

Добавляет монету в отслеживание.

Пример:

```bash
range add BTC
range add ETH --mode balanced --timeframe 4h --lookback 90
```

Что должна делать команда:

* проверить корректность тикера;
* записать монету в хранилище;
* установить базовые настройки;
* при необходимости сразу выполнить первичный расчет.

---

#### `range remove <symbol>`

Удаляет монету из отслеживания.

Пример:

```bash
range remove BTC
```

---

#### `range list`

Показывает список всех монет, которые отслеживаются.

Для каждой монеты желательно показывать:

* символ;
* текущий режим;
* активный диапазон;
* статус;
* дату последней проверки.

---

#### `range show <symbol>`

Показывает подробную информацию по одной монете.

Пример:

```bash
range show BTC
```

В выводе желательно отображать:

* текущую цену;
* активный диапазон;
* рекомендуемый диапазон;
* расстояния до границ;
* статус;
* рекомендацию;
* параметры расчета.

---

### 7.2. Расчет и проверка

#### `range check`

Проверяет все монеты и их активные диапазоны.

Пример:

```bash
range check
```

Должна:

* получить свежие цены;
* пересчитать метрики;
* оценить актуальность диапазонов;
* вывести таблицу статусов;
* сохранить результаты в журнал.

---

#### `range check <symbol>`

Проверяет одну монету.

---

#### `range recalc <symbol>`

Пересчитывает рекомендуемый диапазон для конкретной монеты.

---

#### `range recalc --all`

Пересчитывает рекомендуемые диапазоны для всех монет.

---

### 7.3. Работа с активным диапазоном

#### `range set-active <symbol> --low X --high Y`

Задает активный диапазон вручную.

Пример:

```bash
range set-active BTC --low 59000 --high 76000
```

---

#### `range clear-active <symbol>`

Удаляет активный диапазон.

---

#### `range sync <symbol>`

В будущем: синхронизация активного диапазона с биржей или конфигом бота.

На первом этапе можно не реализовывать.

---

### 7.4. Конфиг и режимы

#### `range config <symbol>`

Показать настройки монеты.

#### `range config set <symbol> key value`

Изменить параметр.

Пример:

```bash
range config set BTC lookback_days 90
range config set BTC mode conservative
```

---

### 7.5. История и журналы

#### `range history <symbol>`

Показать историю проверок и изменений диапазона.

#### `range logs`

Показать последние события системы.

## 8. Логика расчета диапазона

Это центральная часть проекта.
На первом этапе важно не пытаться сделать “идеальный” алгоритм, а сделать понятный, тестируемый и расширяемый движок.

---

## 9. Базовая концепция диапазона

Любой диапазон состоит из:

* **центра**;
* **ширины**;
* **границ**.

Где:

* нижняя граница = центр - половина ширины;
* верхняя граница = центр + половина ширины.

Но на практике центр и ширина должны вычисляться отдельно.

---

## 10. Как выбирать центр диапазона

Нужно поддержать несколько вариантов.

### 10.1. Центр = текущая цена

Самый простой вариант.

Плюсы:

* максимально близко к рынку;
* легко понять.

Минусы:

* сетка может слишком часто смещаться вслед за шумом.

---

### 10.2. Центр = сглаженная цена

Например:

* SMA;
* EMA;
* средняя цена за N свечей;
* медианная цена за период.

Плюсы:

* меньше шума;
* диапазон реже дергается.

Минусы:

* может немного отставать от рынка.

---

### 10.3. Центр = адаптивная средняя

Это вариант на будущие версии, когда центр учитывает тренд и волатильность.

---

## 11. Как выбирать ширину диапазона

Нужно поддержать как минимум несколько стратегий.

### 11.1. Ширина по ATR

Использовать Average True Range как меру волатильности.

Формула может быть примерно такой:

* ширина = ATR × коэффициент × множитель горизонта

Плюсы:

* адаптируется к характеру монеты;
* хорошо работает на волатильном рынке.

---

### 11.2. Ширина по историческому диапазону

Использовать максимум и минимум за период.

Пример:

* берем 60 или 90 дней;
* строим диапазон вокруг центра с учетом исторических экстремумов.

Плюсы:

* интуитивно понятно.

Минусы:

* очень чувствительно к выбросам.

---

### 11.3. Ширина по стандартному отклонению

Использовать волатильность доходностей.

Плюсы:

* ближе к вероятностной модели.

Минусы:

* сложнее интерпретировать.

---

### 11.4. Гибридная ширина

Предпочтительный вариант для развития:

* часть ширины от ATR;
* часть от исторического диапазона;
* ограничение минимальной и максимальной ширины.

---

## 12. Практические ограничения на диапазон

Нужно обязательно ввести защитные правила.

### 12.1. Минимальная ширина

Чтобы программа не предлагала слишком узкий диапазон.

Пример:

* не меньше 8% для спокойных монет;
* не меньше 12–15% для более резких.

---

### 12.2. Максимальная ширина

Чтобы диапазон не становился слишком далеким от цены.

Пример:

* не больше 35% или 50%, в зависимости от режима.

---

### 12.3. Ограничение на удаление центра от рынка

Даже если диапазон формально широкий и устойчивый, его центр не должен слишком далеко отстоять от текущей цены.

Например:

* если текущая цена ушла от центра более чем на 15–20%, диапазон считается устаревшим.

Это один из ключевых критериев проекта.

---

## 13. Режимы расчета

Нужно изначально заложить режимы.

### 13.1. Conservative

Цель:

* максимальная долговечность;
* редкие перестановки;
* допускается более широкий диапазон.

Подходит для:

* BTC;
* ETH;
* более спокойных сценариев.

---

### 13.2. Balanced

Цель:

* разумный компромисс между долговечностью и актуальностью.

Это должен быть режим по умолчанию.

---

### 13.3. Aggressive

Цель:

* держать сетку ближе к цене;
* чаще пересчитывать диапазон;
* выше риск необходимости ручной перестановки.

---

## 14. Статусы диапазона

Программа должна работать не только как калькулятор, но и как система оценки.

Предлагаемые статусы:

### `OK`

* диапазон актуален;
* цена находится в комфортной зоне;
* перестановка не требуется.

### `WARNING`

* цена близко к одной из границ;
* диапазон еще рабочий, но за ним нужно наблюдать.

### `STALE`

* диапазон еще формально рабочий, но уже заметно устарел;
* цена слишком ушла от центра;
* новый диапазон уже выглядит значительно лучше.

### `REPOSITION`

* диапазон рекомендуется переставить;
* текущая сетка не соответствует рыночной ситуации.

### `OUT_OF_RANGE`

* цена вышла за границы диапазона;
* сетка фактически перестала соответствовать рынку.

---

## 15. Критерии оценки диапазона

Нужно заранее определить формальные условия.

### 15.1. Базовые метрики

Для каждой монеты считать:

* текущую цену;
* центр активного диапазона;
* ширину активного диапазона;
* расстояние до верхней границы;
* расстояние до нижней границы;
* отклонение текущей цены от центра;
* ширину рекомендуемого диапазона;
* разницу между активным и рекомендуемым диапазоном.

---

### 15.2. Примеры критериев

#### `OK`, если:

* цена уверенно внутри диапазона;
* до границ осталось достаточно места;
* отклонение от центра не превышает допустимого уровня.

#### `WARNING`, если:

* до одной из границ осталось меньше заданного процента;
* но цена еще внутри диапазона.

#### `STALE`, если:

* цена сильно ушла от центра;
* но формально еще не вышла за пределы диапазона.

#### `REPOSITION`, если:

* рекомендуемый центр сильно отличается от активного;
* или диапазон уже не соответствует текущей волатильности;
* или активный диапазон слишком далеко от рынка.

#### `OUT_OF_RANGE`, если:

* текущая цена выше верхней границы;
* или ниже нижней границы.

---

## 19. Хранилище

### Вариант 1: JSON

Плюсы:

* быстро начать;
* просто отлаживать вручную.

Минусы:

* неудобно расширять;
* хуже история;
* сложнее фильтровать и анализировать данные.

### Вариант 2: SQLite

Рекомендуется как основа.

Плюсы:

* достаточно просто;
* локально;
* удобно хранить историю;
* легко расширять.

Решение:

* для MVP допустим JSON;
* но лучше сразу строить под SQLite.

## 20. Источник рыночных данных

Нужно отдельным слоем выделить получение данных.

Программа должна уметь получать:

* текущую цену;
* исторические свечи;
* при необходимости дневной ATR и другие метрики.

Источники можно подключать поэтапно:

* сначала одна биржа;
* затем абстракция под несколько бирж.

Важно:
слой данных должен быть отделен от логики расчета диапазона.

## 22. Внутренние модули

### 22.1. CLI

Отвечает за прием команд из терминала.

Должен:

* валидировать ввод;
* вызывать сервисы;
* красиво выводить результат.

---

### 22.2. Market Data Service

Отвечает за загрузку цен и истории.

Должен:

* получать текущую цену;
* получать свечи;
* обрабатывать ошибки API;
* кешировать запросы при необходимости.

---

### 22.3. Range Engine

Главный вычислительный модуль.

Должен:

* определить центр;
* определить ширину;
* применить ограничения;
* вернуть рекомендуемый диапазон.

---

### 22.4. Evaluator

Оценивает активный диапазон.

Должен:

* сравнить активный и рекомендуемый диапазоны;
* вычислить метрики;
* присвоить статус;
* сформировать рекомендацию.

---

### 22.5. Storage Layer

Хранит данные о монетах, диапазонах и проверках.

---

### 22.6. Reporter

Формирует красивый вывод в терминал.

На первом этапе:

* таблицы;
* читаемые подписи;
* краткие статусы.

## 23. Алгоритм работы команды `check`

Когда пользователь запускает:

```bash
range check
```

программа должна делать следующее:

1. Загрузить список всех активных монет.
2. Для каждой монеты получить текущую цену.
3. При необходимости загрузить исторические свечи.
4. Пересчитать рекомендуемый диапазон.
5. Считать активный диапазон из хранилища.
6. Посчитать метрики расстояния и отклонения.
7. Определить статус.
8. Сформировать текстовую рекомендацию.
9. Сохранить результат проверки в журнал.
10. Вывести сводку по всем монетам.

---

## 24. Алгоритм работы команды `recalc`

Когда пользователь запускает:

```bash
range recalc BTC
```

программа должна:

1. Получить параметры монеты.
2. Загрузить исторические данные.
3. Рассчитать центр.
4. Рассчитать волатильность.
5. Определить ширину диапазона.
6. Применить минимальные и максимальные ограничения.
7. Сохранить рекомендуемый диапазон.
8. Вывести результат в терминал.

## 27. Требования к качеству кода

Код должен быть:

* разделен по модулям;
* покрыт тестами в критических частях;
* без смешивания CLI, бизнес-логики и хранилища;
* понятным и расширяемым;
* с нормальными именами функций и классов.

Нельзя:

* писать всю логику в `cli.py`;
* смешивать API-запросы и расчет диапазонов в одном месте;
* захардкодить всю конфигурацию без возможности расширения.

## 29. Главные риски проекта

### 29.1. Слишком сложный алгоритм слишком рано

Риск:

* застрять в теории и не дойти до рабочего инструмента.

Решение:

* сначала сделать простой и объяснимый алгоритм.

---

### 29.2. Недостаточно формальные критерии “слишком далеко”

Риск:

* программа будет давать непоследовательные рекомендации.

Решение:

* заранее определить числовые пороги и потом тестировать их.

---

### 29.3. Смешение ролей программы

Риск:

* инструмент для анализа начнет превращаться в торговый движок слишком рано.

Решение:

* на старте оставить программу именно аналитическим менеджером диапазонов.

---

### 29.4. Избыточная универсальность

Риск:

* попытка сразу поддержать все биржи, все режимы и все сценарии.

Решение:

* начать с одного источника данных и 1–2 алгоритмов.
